## Why deep learning works

One fact — actually, it's a mathematical theory you can prove — is that if you could find the shortest program that does very, very well on your data, then you will achieve the best generalization possible. With a little modification you can turn this into a precise theorem. On an intuitive level it's easy to see why it should be the case: if you have some data and you're able to find a shorter program that generates it, then you've essentially extracted all conceivable regularity from that data into your program, and you can use it to make the best predictions possible. If you have data that's so complex that there's no way to express it as a shorter program, then your data is totally random — there's no way to extract any regularity from it whatsoever.

There's a little-known mathematical theory behind this, and the proofs aren't even that hard. But the one slight disappointment is that it's not actually possible — at least given today's tools and understanding — to find the best short program that explains or generates or solves your problem given your data. This problem is computationally intractable. The space of all programs is a very nasty space: small changes to your program result in massive changes in its behavior, as it should be. It makes sense — you have a loop, you change the inside of the loop, and of course you get something totally different. So the space of programs is so hard that, at least given what we know today, search seems completely off the table.

Well, if we give up on short programs, what about small circuits? It turns out we're lucky: when it comes to small circuits, you can just find the best small circuit that solves the problem using back-propagation. This is the miraculous fact on which the rest of AI stands. Once you have a circuit and you impose constraints on it using data, you can find a way to satisfy those constraints — by iteratively making small changes to the weights of your neural network until its predictions satisfy the data. What this means is that back-propagation is extremely profound: it is circuit search. We can't always solve it, but we can solve it sometimes — at those times when we have a practical dataset. It's easy to design artificial datasets for which you cannot find the best neural network, but in practice that seems to not be a problem.

You can think of training a neural network as solving a system of equations, where you have a large number of equation terms like *f(xᵢ, θ) = yᵢ*. Your parameters represent all your degrees of freedom, and you use gradient descent to push the information from these equations into the parameters to satisfy them all. You can see that a neural network — say, one with 50 layers — is basically a parallel computer that's given 50 time steps to run, and you can do quite a lot with 50 time steps of a very powerful, massively parallel computer. For example, I think it is not widely known that you can learn to sort *n* *n*-bit numbers using a modestly sized neural network with just two hidden layers. That's not self-evident, especially since we've been taught that sorting requires *log n* parallel steps — yet with a neural network you can sort using only two parallel steps. So something interesting is going on. These are parallel steps of threshold neurons, so they're doing a little more work — that's the answer to the mystery. But if you've got 50 such layers, you can do quite a bit of logic, quite a bit of reasoning, all inside the neural network.

And that's why it works: given the data, we're able to find the best neural network, and because the network is deep — because it can run computation inside its layers — the best neural network is worth finding. That's really what you need: a model class that's worth optimizing, but also optimizable. Deep neural networks satisfy both constraints, and this is why everything works. This is the basis on which everything else resides.

## Reinforcement learning

Now I want to talk a little about reinforcement learning. Reinforcement learning is a framework for evaluating agents on their ability to achieve goals in complicated, stochastic environments. You've got an agent plugged into an environment, as shown in the figure, and for any given agent you can simply run it many times and compute its average reward. The interesting thing about the RL framework is that there exist interesting, useful RL algorithms. The framework existed for a long time; it became interesting once we realized that good algorithms exist. They're not perfect algorithms, but they're good enough to do interesting things. The mathematical problem is one where you need to maximize the expected reward.

Now, one important way in which the reinforcement learning framework is not quite complete is that it assumes the reward is given by the environment. In this picture, the agent sends an action, and the environment sends back an observation and a reward. The way this is not the case in the real world is that we figure out what the reward is from the observation — we reward ourselves. The environment doesn't say "hey, here's some negative reward"; it's our interpretation of our senses that lets us determine the reward. And there is only one real, true reward in life — existence or nonexistence — and everything else is a corollary of that.

So what should our agent be? You already know the answer: it should be a neural network, because whenever you want to do something dense, it's going to be a neural network. You want the agent to map observations to actions, so you parametrize it with a neural net and apply a learning algorithm. Let me explain how reinforcement learning works — this is model-free reinforcement learning. It's been used in practice everywhere; it's very robust and very simple, but it's also not very efficient. Here's the one-sentence description of what happens: try something with a bit of added randomness, and compare the result to your expectation. If the result surprises you — if it exceeds your expectation — then change your parameters to take those actions in the future. That's it. That's the whole idea of reinforcement learning: try it out, see if you like it, and if you do, do more of that in the future. It's not difficult to formalize mathematically, but this is really what's going on.

In a regular neural network you might say: you run the network, get an answer, compare it to the desired answer, and whatever difference you have, you send it back to change the network — that's supervised learning. In reinforcement learning, you run the network, add a bit of randomness to your action, and then, if you like the result, your randomness turns into the desired target, in effect. So that's it — trivial.

The math exists; I won't explain what these equations mean — the point isn't to derive them but just to show that they exist. There are two classes of RL algorithms. One is the **policy gradient**, where you take the sum of expected rewards, crunch through the derivatives, expand the terms, do some algebra, and you get a derivative that — miraculously — has exactly the form I told you: try some actions, and if you like them, increase the log-probability of those actions. That truly follows from the math. It's very nice when the intuitive explanation has a one-to-one correspondence to what you get in the equation. There's a different class of RL algorithms that's a bit harder to explain, called **Q-learning**-based algorithms. They're a bit less stable, a bit more sample-efficient, and they have the property that they can learn not only from the data generated by the actor but from any other data as well. So they have a different robustness profile, which will be a little important later — but it's only going to be a technicality. This is the on-policy / off-policy distinction. It's a bit technical, so if you find it hard to understand, don't worry about it; if you already know it, you already know it.

So what's the potential of reinforcement learning? Why should we be excited about it? There are two reasons. The RL algorithms of today are already useful and interesting, especially if you have a really good simulation of your world — you could train agents to do lots of interesting things. But what's really exciting is if you can build a super-sample-efficient RL algorithm: you give it a tiny amount of data, and the algorithm crunches through it and extracts every bit of entropy in order to learn in the fastest way possible. Today our algorithms are not particularly efficient — they're data-inefficient — but as the field keeps making progress, this will change.

## Meta-learning

Next I want to dive into meta-learning. Meta-learning is a beautiful idea that doesn't really work — but it kind of works, and it's really promising. So what's the dream? We have learning algorithms; perhaps we could use them to *learn to learn*. How would you do that? You take a system and train it not on one task but on many tasks, and you ask it to learn to solve these tasks quickly. That may actually be enough.

Here's how most traditional meta-learning works. You have a model — a big neural network — and instead of training *cases* you have training *tasks*, and instead of test cases you have test tasks. So your input, instead of being just your current test case, would be all the information about the new task plus the test case, and you try to output the prediction for that test case. Basically you say: "I'm going to give you your ten examples as part of your input to the model — figure out how to make the best use of them." It's a really straightforward idea: you turn the neural network into the learning algorithm by turning a training task into a training case. That's meta-learning in one sentence.

There have been several success stories which I find very interesting. One is learning to recognize characters quickly: there's a dataset produced at MIT by Lake et al. (Omniglot) with a large number of different handwritten characters, and people have been able to train extremely strong meta-learning systems on it. Another very successful example is neural architecture search by Zoph and Le from Google, where they found a neural architecture that solved one small problem well, and then it generalized so that it successfully solved large problems too. That's the "small number of bits" kind of meta-learning, where you learn the architecture — or maybe even learn a small program or learning algorithm — which you then apply to new tasks. So that's the other way of doing meta-learning. But the point is: what's really happening in meta-learning, in most cases, is that you turn a training task into a training case and pretend this is totally normal deep learning. That's the entirety of meta-learning; everything else is minor details.

## Hindsight Experience Replay

Now that I've finished the introduction, I want to discuss work by different people from OpenAI, starting with **Hindsight Experience Replay**. This was a large effort by Andrychowicz et al. to develop an RL algorithm that doesn't solve just one task but many tasks, and that learns to make use of its experience in a much more efficient way.

I want to discuss one problem in reinforcement learning — actually a set of related problems. One really important thing you need to learn to do is **explore**. You start out in an environment, you don't know what to do — so what do you do? One very important thing that has to happen is that you must get rewards from time to time. If you try something and you never get rewards, how can you learn? That's the crux of the problem. And relatedly: is there any way to meaningfully benefit from your experience — from your failures? If you try to achieve a goal and fail, can you still learn from it?

The idea is: instead of asking your algorithm to achieve a single goal, you want to learn a policy that can achieve a very large family of goals. For example, instead of reaching one state, you want to learn a policy that reaches every state of your system. What's the implication? Anytime you do something, you achieve *some* state. Suppose you say "I want to achieve state A," you try your best, and you end up in state B. You can either conclude, "well, that was disappointing, I haven't learned anything, I still have no idea how to achieve state A" — or, alternatively, you can say, "wait a second, I've just reached a perfectly good state, B. Can I learn how to achieve state B from my attempt to achieve state A?" And the answer is yes, you can — and it just works.

I want to point out a small subtlety here, which may be interesting to those very familiar with the distinction between on-policy and off-policy. When you try to achieve A, you're doing on-policy learning for reaching state A, but you're doing off-policy learning for reaching state B — because you'd take different actions if you were actually trying to reach B. That's why it's very important that the algorithm you use here can support off-policy learning. But that's a minor technicality. The crux of the idea is: you make the problem easier by ostensibly making it harder — by training a system that aspires to reach every state, to achieve every goal, to master its environment in general. You build a system that always learns something — from success as well as from failure — because if it tries to do one thing and does something else, it now has training data for how to achieve that something else.

Let me show you a video of how this works in practice. One challenge in RL is the need to **shape the reward**. At the start of learning, the system doesn't know much and probably won't achieve your goal, so it's important to design your reward function to give gradual, smooth, continuous increments — so that even when the system isn't very good, it makes progress. If instead you give a very **sparse** reward — achieved only when you reach the final state — it becomes very hard for normal RL algorithms, because you never get the reward, and no reward means no learning. But here, because you learn from failure as well as success, this problem simply doesn't occur. So this is nice — let's look at the videos a bit more. It's nice how it confidently and energetically moves the little green puck to its target. And here's another one.

Okay, we can skip ahead — it works on physical robots as well, but we can skip it. The point is that the Hindsight Experience Replay algorithm is directionally correct, because you want to make use of all your data, not just a small fraction of it. Now, one huge question is: where do you get the high-level states? In the work I've shown so far, the system is asked to achieve low-level states. So I think one thing that will become very important for these approaches is **representation learning** and **unsupervised learning** — figuring out the right states, the right state space of goals that's worth achieving.

## Sim-to-real with meta-learning

Now I want to go through some real meta-learning results, and I'll show you a very simple way of doing **sim-to-real** — from simulation to the physical robot — with meta-learning. This was a really nice intern project in 2017. In the domain of robotics, it would be nice if you could train your policy in simulation and have that knowledge carry over to the physical robot. We can build simulators that are okay, but they can never perfectly match the real world unless you want an insanely slow simulator. So your simulation is just not going to match reality; there will be some resemblance, but that's it. How can we address this?

Let me show you one simple idea. One thing that would be nice is if you could learn a policy that would quickly adapt itself to the real world. If you want a policy that can quickly adapt, you need to make sure it has opportunities to adapt during training. So instead of solving the problem in just one simulator, we add a huge amount of variability to the simulator: we randomize the friction, the masses, the lengths and dimensions of the different objects. You randomize the physics, simulate in lots of different ways — and, importantly, you don't tell the policy how you randomized it. So what's it going to do? You take your policy and put it in an environment, and it says "well, this is really tough — I don't know what the masses are, I don't know what the frictions are, I need to try things out and figure out the friction from the responses I get from the environment." ## Learning a hierarchy of actions

I want to discuss one more application of meta-learning: learning a hierarchy of actions. Now, no one has been able to get real value-add from hierarchical RL yet — all the really convincing RL results so far do not use it — because we haven't quite figured out the right way to do hierarchical RL. ## Limitations of meta-learning

Now I want to spend one slide addressing the limitations of high-capacity meta-learning. The specific limitation is that the training-task distribution has to be equal to the test-task distribution. I think this is a real limitation, because in reality the new tasks you want to learn are in some ways fundamentally different from anything you've seen so far. For example, when you go to school you learn lots of useful things, but then you go to work and only a fraction of what you learned carries over — you need to learn quite a few more things from scratch. Meta-learning would struggle with that, because it really assumes the distribution over training tasks equals the distribution over test tasks. That's the limitation. I think that as we develop better algorithms for being robust when the test tasks are outside the distribution of the training tasks, meta-learning will work much better.

## Self-play

Now I want to talk about self-play. Self-play is a very cool topic that's starting to get attention only now. I want to start by reviewing very old work called **TD-Gammon**, all the way from 1992 — so it's 26 years old now — done by Gerald Tesauro. This work is incredible because it has so much relevance today. What they did, basically, was take two neural networks and let them play backgammon against each other, and train them that way. It's a super-modern approach — you'd think it was a paper from 2017 — except that when you look at this plot, it shows you only have 10, 20, 40, and 80 hidden units for the different colors, and you notice that the largest neural network works best. So in some ways not much has changed, and this is the evidence. In fact, they were able to beat the world champion in backgammon, and they discovered new strategies that the best human backgammon players had never noticed — and it was determined that the strategy discovered by TD-Gammon was actually better. So that's pure self-play with Q-learning, which then remained dormant until the DQN work on Atari by DeepMind.

Other examples of self-play include **AlphaGo Zero**, which learned to beat the world champion at Go without using any external data whatsoever. Another result in this vein is from OpenAI — our **Dota 2 bots**, which were able to beat the world champion at the 1v1 version of the game.

I want to spend some time talking about the allure of self-play and why I think it's exciting. One important problem we must face as we try to build truly intelligent systems is: what is the task? What are we actually teaching the systems to do? One very attractive attribute of self-play is that the agents create the environment: by virtue of the agents acting in the environment, the environment becomes difficult for the other agents. Here's an example of an iguana interacting with snakes that try to eat it — unsuccessfully this time. The iguana strains and escapes, and the fact that you have this arms race between the snakes and the iguana motivates their development, potentially without bound. Interesting work in this direction was done in 1994 by Karl Sims — there's a really cool video on YouTube by Karl Sims you should check out — which shows the work he's done. Here you have a competition between agents where you evolve both their behavior and their morphology, and the agents are trying to gain possession of a green cube. You can see that the agents create the challenge for each other, and that's why they need to develop.

So one thing we did — this is work by Bansal et al. from OpenAI — we said, okay, can we demonstrate some unusual results in self-play that would really convince us there's something there? What we did is create a small ring with two humanoid figures whose goal is just to push each other outside the ring. They don't know anything about wrestling, about standing, about balance, about centers of gravity — all they know is that if you don't do a good job, your competition is going to do a better job. One of the really attractive things about self-play is that you always have an opponent that's roughly as good as you are. To learn, you need to sometimes win and sometimes lose — you can't always win, you must sometimes fail and sometimes succeed. So let's see what happens — the green humanoid was able to block the other. In a well-balanced self-play environment, the competition is always level no matter how good or bad you are; you have a competition that provides exactly the right challenge.

One more thing here: this video shows transfer learning. So the biggest open question with this research is: how do you learn agents in a self-play environment such that they do whatever they do, but are then able to solve a battery of tasks that's useful for us — that's explicitly specified externally?

## The endgame: a society of agents

I also want to talk about the endgame of the self-play approach. One thing we know about the human brain is that it increased in size fairly rapidly over the past two million years. My theory for why this happened is that our ancestors reached a point where the thing most important for your survival is your standing in the tribe, rather than the tiger and the lion. Once the most important thing is how you deal with those other large-brained beings, then it really helps to have a slightly larger brain — and I think that's what happened. So I think what should happen, if we successfully follow this path, is that you should create a society of agents which will have language, theory of mind, negotiation, social skills, trade, economy, politics, a justice system — all these things should happen inside the multi-agent environment. Now I want to make a speculative digression. **Q:** Neural networks are bio-inspired, but back-propagation doesn't look like what's going on in the brain — signals in the brain go one direction, down the axons, whereas back-propagation requires errors to be propagated back up the wires. **A:** That's an extremely good question. One place we can get inspiration from is the evolution of cooperation. We cooperate ultimately because it's much better for the person to be cooperative than not. I think what should happen is that, if you have a sufficiently open-ended game, cooperation will be the winning strategy — so we'll get cooperation whether we like it or not.

**Q:** You mentioned the complexity of simulating friction. I was wondering if you feel there exist open complexity-theoretic problems relevant to AI, or whether it's just a matter of finding good approximations to the types of problems humans tend to solve.

**A:** At a very basic level, we know that whatever algorithm we run is going to run fairly efficiently on some hardware, so that puts a pretty strict upper bound on the true complexity of the problems we're solving. By definition, we're solving problems that aren't too hard in a complexity-theoretic sense. It is also the case that, while the overall thing we do is not hard in that sense — and indeed humans cannot solve NP-complete problems in general — many of the optimization problems we pose to our algorithms are intractable in the general case, starting from neural net optimization itself. It's easy to create a family of datasets for a neural network with a very small number of neurons such that finding the global optimum is NP-complete. How do we avoid it? We just try gradient descent anyway, and somehow it works. But without question, we do not solve problems that are truly intractable. I hope that answers the question.

**Q:** It seems like an important sub-problem on the path to AGI will be understanding language, and the state of generative language modeling right now is pretty abysmal. What do you think are the most productive research trajectories toward generative language models?

**A:** First, you're completely correct that the situation with language is still far from great. But progress has been made even without any particular innovations: simply scaling up the models that exist today on larger datasets is going to go surprisingly far — not even larger datasets, but larger and deeper models. For example, if you trained a language model that's a thousand layers, I think it's going to be a pretty amazing language model. We don't have the cycles for it yet, but I think that will change very soon. I also agree there are some fundamental things missing in our current understanding of deep learning that prevent us from really solving the problem. One thing that seems patently wrong is that we train a model, then stop and freeze it — even though it's the training process where the magic really happens. The training process is the truly general part of the whole story, because TensorFlow code doesn't care which dataset to optimize — it just says "whatever, give me the dataset, I'll solve them all." That ability feels really special, and we're not using it at test time. It's hard to speculate about things you don't know the answer to, but all I'll say is: simply training bigger, deeper language models will go surprisingly far — and also doing things like training at test time, inference at test time, would be another important boost to performance.

**Q:** It seems like another interesting approach to solving RL problems could be to go for the evolutionary route, using evolutionary strategies — though they have their caveats. Is OpenAI working on something related, and what's your general opinion on them?

**A:** At present, I believe evolutionary strategies are not great for reinforcement learning — normal RL algorithms, especially with big policies, are better. But if you want to evolve a small, compact object — like a piece of code, for example — I think that would be a place worth seriously considering. Evolving a beautiful piece of code is a cool idea that hasn't been done yet, so there's still a lot of work to be done before we get there.

**Q:** You mentioned that the right goal is a political problem — can you elaborate on that, and what you think our approach to getting there would be?

**A:** I can't comment too much, because we now have a few people thinking about this full-time at OpenAI, and I don't have a strong enough opinion to say anything too definitive. At a very high level: given the size of the impact — if you go into the future, whenever it happens, when you build a computer that can do anything better than a human (and it will happen, because the brain is physical) — the impact on society is going to be completely massive and overwhelming. It's very difficult to imagine even if you try really hard. What it means is that there will be many people who care about this strongly, and as the impact increases gradually — with self-driving cars, more automation — I think we'll see a lot more people care.

**Q:** Do we need a very accurate model of the physical world, simulated, in order to have agents that can eventually come into the real world and do something approaching human-level intelligence tasks?

**A:** Very good question. If that were the case, we'd be in trouble, and I'm very certain it can be avoided. The real answer has to be that you learn the *problem*: you learn to negotiate, you learn to persist, you learn lots of different useful life lessons in the simulation — and yes, you learn some physics too — but then you go out into the real world and have to start over to some extent, because many of your deeply held assumptions will be false. That's one reason I care so much about never stopping training: you've accumulated your knowledge, then you go into an environment where some of your assumptions are valid, and you continue training and try to connect the new data to your old data. This is an important requirement for our algorithms — already met to some extent, but it will have to be met a lot more — so that you can take the partial knowledge you've acquired, go into a new situation, and learn some more. Literally the example of going to school: you learn useful things, then you go to work; your four years of CS undergrad won't fully prepare you for whatever you need to know at work, but it will help somewhat — you'll be able to get off the ground, but there will be lots of new things to learn. That's the spirit of it.

**Q:** One of the things you mentioned early in your talk is that one limitation of this style of reinforcement learning is that there's no self-organization — you have to tell it when it did a good thing or a bad thing. That's also a problem in neuroscience: when you teach a rat to navigate a maze, you have to artificially tell it what to do. So where do you see the research moving forward to introduce this notion of self-organization?

**A:** Without question, one really important thing you need is to be able to infer the goals and strategies of other agents by observing them. That's a fundamental skill we need to embed into the agent. For example, if you have two agents and one is doing something, the other says "well, that's really cool, I want to be able to do that too," and goes and does it. I'd say that this is a very important component: in terms of reward, you see what they do, you infer the reward, and now we have a knob that says "you see what they're doing — now go and try to do the same thing." As far as I know, this is one of the important ways in which humans are quite different from other animals — in the scale and scope with which we copy the behavior of other humans.

**Q:** Quick follow-up: it's kind of obvious how that works in the scope of competition, but what about arbitrary tasks — like I'm in a math class and I see someone doing a problem a particular way and I think "that's a good strategy, maybe I should try that out"? How does that work in a non-competitive environment?

**A:** I think that's going to be a little bit separate from the competitive environment, but it will have to be baked in — or maybe evolved into — the system either way. If you have other agents doing things, they're generating data that you observe, and the only way to truly make sense of the data you see is to infer the goal of the agent, their strategy, their belief state. That's important also for communicating with them: if you want to successfully communicate with someone, you have to keep track of both their goal and their belief state, their knowledge. So I think you'll find there are many connections between understanding what other agents are doing, inferring their goals, imitating them, and successfully communicating with them.
